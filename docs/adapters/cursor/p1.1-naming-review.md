# Bridge Cursor Adapter — P1.1 Naming Review

**Status:** P1.1 (Empfehlungen only — **keine Umbenennung**)  
**Datum:** 2026-05-27  
**Scope:** `cursor.domain.*`, `bridge.ui.*`, `channel.*`, `cursor.surface.*`, `cursor.subsystem.*`

---

## 1. Bewertungskriterien

| Kriterium | Bedeutung |
|-----------|-----------|
| **Stabil P1** | Für Cursor-Adapter-Dokumentation und P2-Start ohne Rename |
| **Adapter-spezifisch** | Gilt primär für ein Zielprogramm (Cursor/VS Code-Fork) |
| **Global wiederverwendbar** | Kann für mehrere Adapter gleiche ID tragen |
| **Vor Promotion ändern** | Rename empfohlen, bevor JSON in Runtime geht |
| **Später** | Erst bei Multi-Adapter-Rollout entscheiden |

### Empfehlung Domain-Naming (A1)

| Strategie | Beschreibung | Empfehlung |
|-----------|--------------|------------|
| **A) `cursor.domain.*` beibehalten** | Einfach, klar erster Adapter | **Kurzfristig P1/P2** |
| **B) `bridge.domain.*` + `adapterId`** | Multi-Adapter-ready | **Vor zweitem Adapter** |
| **C) Hybrid** | Bridge-Cross-Cutting → `bridge.domain.*`; IDE → `cursor.domain.*` | **Pragmatisch vor Promotion** |

**P1.1-Entscheidung:** Kein Rename. Hybrid (C) als **Ziel vor Runtime-Promotion** dokumentieren.
